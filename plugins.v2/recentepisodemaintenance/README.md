# 最近剧集维护

## 插件说明

定时维护 MoviePilot 最近整理入库的 Jellyfin 剧集。筛选依据是 MoviePilot 整理历史中的整理时间，不使用剧集的实际发布日期。

### 1. 刷新最近整理剧集元数据

查询 MoviePilot 最近 N 天内成功整理的剧集记录，在 Jellyfin 中定位对应视频，并比较 Jellyfin 当前标题与 MoviePilot 按当前数据和命名规则生成的整理预览标题。标题不一致时执行刷新，标题一致时跳过，用于处理以下情况：

- 新剧入库时 TMDb 标题、简介或图片尚未更新；
- 旧剧重新入库后，Jellyfin 因网络问题产生了不完整或错误的刮削结果。

刷新模式与 Jellyfin 界面一致：

- 扫描新的和有修改的文件；
- 搜索缺少的元数据；
- 覆盖所有元数据（默认）。

“覆盖所有元数据”会覆盖现有元数据。“替换现有图片”默认开启，可以单独关闭；扫描模式不执行覆盖操作。媒体库启用 NFO 保存后，Jellyfin 会按自身设置更新 NFO 文件。

同时启用元数据刷新和重新整理时，标题不一致的剧集本轮只提交 Jellyfin 刷新，并暂缓重新整理。后续运行再次确认标题一致后，才会执行重新整理；重新整理前还会重新获取一次安全预览。重新整理后若 Jellyfin 又产生错误标题，后续运行会再次检测并刷新。

插件会保存每条整理记录的处理状态和最近一次整理预览。刷新成功后等待 30 分钟再确认，避免 Jellyfin 尚未完成处理时重复提交刷新；刷新或重新整理过的记录会在后续运行中优先复查，确认 Jellyfin 标题一致且按当前规则生成的文件路径不再变化后，该记录才处理完成。尚未发生过刷新或重新整理的正常记录按 24、48、72 小时逐步降低复查频率，以发现 TMDb 后续补充的正式标题；到期复查和新记录会交替进入检查范围，避免其中一类长期等待。最近 N 天只限制新记录进入队列，已经进入队列的记录即使超过该范围也会保留到实际完成，不会因单轮检查数量不足而丢失；超过范围的正常记录在轮到最终检查并确认无变化后完成。待复查记录直接使用已保存的预览，不会重复请求 MoviePilot 生成预览。为避免异常命名规则反复修改文件，连续重新整理两次后预览路径仍变化时会停止自动处理，并在日志中提示人工检查。

“单次最大处理数量”限制每轮元数据刷新与重新整理的合计操作数。标题一致、命名未变化等无需操作的记录不占用该数量；为避免一次运行查询过多数据，每轮最多检查该数量五倍的整理记录。达到操作上限后，尚未执行的记录会保存为待复查并在下一轮优先继续。

“重新处理需人工检查记录”是一次性开关。保存配置后，插件会按失败阶段重置对应重试次数并立即重新入队：附件生成失败继续等待附件，旧附件清理失败继续执行清理，其他记录重新从刷新或重新整理阶段检查。开关保存后自动关闭。

配置项：执行周期，最近 N 天，媒体服务器，维护媒体库，刷新模式，是否替换现有图片，重新处理需人工检查记录，单次最大处理数量。

### 2. 重命名最近整理剧集文件

查询同一批 MoviePilot 整理历史记录，并调用 MoviePilot 自身的历史记录重新整理功能。整理方式、目标存储和刮削流程沿用原整理记录，不直接修改文件名。同一集同时存在视频和字幕等整理记录时，以视频记录作为判断基准，并记录同批附件历史；普通字幕只有在确认新名称对应文件已经生成后才会清理旧名称文件。

用于文件命名模板包含剧集标题的情况，例如新剧刚发布时文件名仍是“第 3 集”，后续 TMDb 更新正式标题后重新整理文件名。

“清理旧名称附件”默认开启。插件会在重新整理前记录与旧视频同名的 NFO、图片、字幕、弹幕和 trickplay 目录；确认新视频、NFO 和图片就绪后延迟清理，再触发 Jellyfin 扫描并复查一次。Jellyfin 扫描提交失败时，相关记录会保留为等待扫描，下一轮先重试扫描，不会提前标记完成。普通字幕只有在新名称对应文件已经存在时才会删除旧文件；XML 弹幕及同名 .danmu.ass 会在调用 MoviePilot 重新整理前改成保护名称，整理成功后再归位到新视频名称，若新名称文件已经生成则删除保护副本。插件不会扫描或删除未被本次重命名记录的其他文件。同一集出现新的下载整理记录时，仍在保护附件、等待清理或等待扫描的旧记录会先完成关键步骤，再处理新记录。

配置项：是否启用重新整理，试运行模式，是否跳过命名未变化的文件，是否清理旧名称附件，单次最大处理数量。

### 3. 试运行模式

试运行模式下只查询 MoviePilot 整理记录、匹配 Jellyfin 条目并生成重新整理预览，不刷新元数据，也不修改文件。

标题不一致的剧集在试运行中同样会暂缓重新整理，用于预览正式运行时“先刷新、后续再整理”的处理顺序。

首次使用建议开启试运行模式，并将单次最大处理数量设置为 1。

## 安全说明

- 新任务只来自最近 N 天内成功的 MoviePilot 剧集整理记录，已入队任务会保留到完成；
- Jellyfin 按剧集目录、相对路径和唯一文件名匹配，不依赖 Jellyfin 当前的标题、TMDB ID 或季集元数据；
- Jellyfin 当前标题与 MoviePilot 当前整理预览中的剧集标题一致时，跳过元数据刷新；
- 不直接重命名主视频文件，由 MoviePilot 按历史整理记录重新执行整理；
- 旧名称附件只按重新整理前保存的精确路径清理，并在新附件就绪后执行；
- MoviePilot 支持整理预览时，会先预览新路径，文件名没有变化时可自动跳过；
- 待刷新和待重命名复查的记录会优先进入后续运行，尚未操作的正常记录按 24、48、72 小时逐步降低复查频率，操作后确认正常的记录不再重复处理；
- 每次运行开始先检查 MoviePilot 整理历史、整理预览和 Jellyfin 服务是否可用，兼容性检查失败时不会继续逐条处理；
- 刷新、重新整理、附件清理和媒体库扫描会在调用外部接口前保存操作意图，调用后立即保存结果；状态保存失败会有限次重试并停止本轮，下一轮根据遗留意图恢复对应阶段；
- 开启通知后，仅在发生刷新、重新整理或失败时发送 MoviePilot 消息，无变化的运行只保留日志；
- 普通单集失败不会中断整次任务；处理状态无法可靠保存时会停止本轮，以避免重复执行外部操作。

## 推荐测试步骤

1. 保持试运行模式，将单次最大处理数量设置为 1，并开启需要测试的功能。
2. 确认日志显示的 MoviePilot 整理记录、Jellyfin 条目和预览文件名正确。
3. 先只测试“刷新最近整理剧集元数据”，确认 Jellyfin 元数据、NFO 和图片符合预期。
4. 再测试“重命名最近整理剧集文件”，确认主视频和字幕、XML 弹幕、NFO、图片等伴随文件均符合预期。
